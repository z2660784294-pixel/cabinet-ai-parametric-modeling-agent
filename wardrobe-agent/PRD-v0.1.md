# 组合柜 AI 建模生产助手 PRD v0.1

| 项目 | 内容 |
| --- | --- |
| 文档版本 | v0.1 |
| 产品定位 | 内部建模提效工具 |
| 目标用户 | 内部建模人员、模型库维护人员、研发/QA |

| 更新时间 | 2026-06-20 |

## 1. 背景与目标

当前组合柜参数化建模需要人工完成需求理解、单元柜拆解、模型选型、布局摆放、参数联动、脚本生成和编辑器验证。流程长、重复性高，对模型库熟悉度要求高，也难以稳定复现和回归。

本产品面向内部建模团队，目标是通过 AI 和工具链把“图片/文字需求”快速转化为可确认、可执行、可校验的参数化组合柜模型，让建模人员从重复拆柜、找模型、写参数和调位置中解放出来，主要承担结构确认和少量修正工作。

一句话定义：

> 面向内部建模人员的组合柜 AI 生产助手，将图片/文字/已有组合柜样本转化为可确认、可执行、可回归验证的参数化组合柜模型。

## 2. 产品定位

本产品不是 C 端设计工具，也不是全屋智能设计系统，而是内部生产工具。

核心价值：

- 提升组合柜从需求到参数化编辑器成品的建模效率。
- 降低对单个建模人员模型库经验的依赖。
- 将 AI 输出纳入人工确认和自动校验闭环。
- 通过样本、模型库和回归体系，让主路径越用越稳定。

## 3. 用户角色

| 角色 | 主要诉求 | 典型任务 |
| --- | --- | --- |
| 建模人员 | 快速把参考图/文字需求转为可编辑组合柜 | 上传需求、确认单元柜和布局、查看生成结果 |
| 模型库维护人员 | 保证单元柜可被 AI 检索和正确调用 | 同步模型库、维护 image profile、补充候选模型 |
| 研发/QA | 保证链路稳定、改动可回归 | 跑 design2edit / abd2edit、分析失败 case |
| 产品/项目负责人 | 衡量提效效果和质量边界 | 跟踪成功率、耗时、人工修改次数、回归通过率 |

## 4. 产品范围

### 4.1 MVP 范围

MVP 聚焦组合柜建模提效，优先支持“一柜到顶衣柜/组合柜”类目。

MVP 包含：

- 图片/文字需求输入。
- AI 生成组合柜 ABD 草稿。
- 从本地模型库匹配单元柜 `obsBrandGoodId`。
- 用户确认单元柜和布局。
- 生成 `design.json`。
- 生成 `cabinet_script.js`。
- 通过 MCP 执行到参数化编辑器。
- 进行实例数量和 bbox 校验。
- 生成可诊断产物和回归报告。

### 4.2 非目标

MVP 不做：

- 面向终端消费者的设计产品。
- 全屋空间布局设计。
- 完全无人审核的一键出图。
- 单元柜内部门板、抽屉、开放格的可视化编辑。
- 材质、把手、工艺参数的精细交互编辑。
- 跨多行业线的泛化建模能力。

## 5. 总体路径关系

产品由 1 条主路径和 2 条辅助路径构成。

```text
主路径：新柜生产
图片/文字需求
  -> AI 生成 ABD 草稿
  -> 用户确认单元柜和布局
  -> 生成 design/script
  -> 编辑器执行
  -> 自动校验

辅助 A：复建与验证
已有 abd/design 或仅 abd
  -> 生成或复用 design
  -> 生成 script
  -> 编辑器执行
  -> editData / bbox / baseline 对比

辅助 B：模型库与样本资产
线上模型库/收藏夹/组合柜样本
  -> 同步商品、参数、预览图、profile、样本
  -> 反哺主路径选型、参数生成和回归质量
```

三者关系：

- 主路径负责“生产一个新柜”。
- 辅助 A 负责“复建、验证、回归、定位问题”。
- 辅助 B 负责“提供模型库、profile、参数和样本资产”。

## 6. 主路径：AI 辅助组合柜建模生产流程

### 6.1 流程概述

```text
输入需求
  -> AI 解构组合柜
  -> 单元柜搜索与选型
  -> ABD 草稿
  -> 人工确认单元柜和布局
  -> 确认版 ABD
  -> 生成 design.json
  -> 生成 cabinet_script.js
  -> MCP 执行
  -> 实例/bbox 校验
  -> 成功或进入 review
```

### 6.2 输入需求

输入可以包含：

- 参考图片。
- 文字描述。
- 整柜尺寸，如宽、高、深。
- 结构偏好，如左边开放格、右边双门柜、顶部收口等。

系统应允许文字和图片任一存在；只有文字时按文字建模，只有图片时按图像解构。

### 6.3 AI 解构组合柜

AI 根据需求判断：

- 组合柜单元柜数量。
- 每个单元柜类型，如高柜/下柜、边柜、开放柜等。
- 单元柜左右/上下位置关系。
- 大致宽、高、深。
- 门数、抽屉、开放格、把手、材质等结构特征。

注意：门数、抽屉、开放格等信息在主路径中主要用于辅助选型，不作为确认页的最终编辑对象。

### 6.4 单元柜搜索与选型

系统基于单元柜特征，从本地模型库中匹配 `obsBrandGoodId`。

选型依据包括：

- 模型名称。
- image profile。
- 门数、抽屉、开放格、边柜方向。
- 单元柜宽高比例。
- 左/右边柜限定。

系统应保留候选模型列表，便于确认页替换。

### 6.5 ABD 草稿

ABD 是主路径的结构化中间产物。MVP 使用扁平 `abd.json` 结构。

核心字段：

- `name`
- `cabinetSize`
- `units[]`
- `units[].id`
- `units[].name`
- `units[].obsBrandGoodId`
- `units[].position`
- `units[].size`
- `units[].cells`
- `units[].candidates`

约束：

- `units[]` 为一维数组。
- `position` 表示单元柜左后下点。
- `size.x/y/z` 分别表示宽、深、高。
- `cells` 用于表达表格布局占位。
- MVP 阶段确认后的 ABD 不应包含 `scale` 和 `rotate`。

### 6.6 人工确认 ABD

ABD 确认页是主路径的正式质量门，不是临时兜底。

确认页定位：

> 确认“哪些单元柜，以什么布局组成组合柜”。

用户可编辑：

- 组合柜宽、高。
- 单元柜布局网格。
- 单元柜位置和尺寸比例。
- 单元格增删。
- 单元格合并、拆分。
- 单元柜模型替换。

用户不可编辑：

- 单元柜内部门板。
- 抽屉数量。
- 开放格数量。
- 把手样式。
- 材质细节。
- 单元柜内部参数。
- 父子参数公式。

产品原则：

- 确认后的 ABD 是生产事实。
- AI 的图像分析结果只是草稿依据。
- 单元柜内部细节由选中的模型承载，不在确认页二次编辑。

### 6.7 生成参数化设计

系统根据确认后的 ABD 生成 `design.json`。

`design.json` 负责表达：

- 组合柜父模型参数。
- 单元柜实例列表。
- 子模型位置公式。
- 子模型尺寸公式。
- 父子参数绑定。
- 材质、收口、镜像等工艺参数。

设计生成应满足：

- `design.units[]` 覆盖 `abd.units[]` 的每一个实例。
- `design.units[].id` 使用 ABD 中真实实例 id。
- 默认参数下，单元柜尺寸和布局应与 ABD 一致。
- 参数表达式优先于写死数值。
- 没有明确要求时不旋转子模型。

### 6.8 生成并执行脚本

系统调用脚本生成器，将 `abd.json + design.json` 转为 `cabinet_script.js`。

执行流程：

1. 清空参数化编辑器场景。
2. 执行 `cabinet_script.js`。
3. 获取编辑器场景信息。
4. 可选导出预览图和 editData。

### 6.9 自动校验

校验包括：

- 实例数量是否一致。
- `obsBrandGoodId` 是否缺失或多出。
- bbox 位置和尺寸是否与 ABD 预期一致。
- 是否存在 only_in_scene / only_in_abd / different。

当 bbox 不一致：

- 将 `bbox_diff.json` 作为反馈。
- 重新生成 `design.json` 和脚本。
- 最多重试 2 次。
- 仍不收敛则保留当前结果并进入人工 review。

## 7. 辅助 A：复建与验证路径

辅助 A 是主路径的验证、复建和回归能力，不是图片理解入口。

### 7.1 辅助 A1：已有 design 验证

输入：

- `abd.json`
- `design.json`
- 可选 baseline `editData.json`

流程：

```text
abd.json + design.json
  -> cabinet_script.js
  -> 编辑器执行
  -> 导出 editData
  -> baseline 对比
  -> 报告
```

用途：

- 验证脚本生成器是否稳定。
- 验证 PMBuilder 执行是否稳定。
- 验证设计改动是否破坏已有 case。
- 排查问题是否发生在 AI 设计生成之前或之后。

### 7.2 辅助 A2：ABD-only 复建

输入：

- `abd.json`

流程：

```text
abd.json
  -> 重新生成 design.json
  -> cabinet_script.js
  -> 编辑器执行
  -> 实例/bbox 校验
  -> 报告
```

用途：

- 验证确认后的 ABD 是否能自动复建。
- 验证 design 生成 agent 的稳定性。
- 支持从已有结构化方案恢复生成结果。

说明：

- 当前脚本生成器不能跳过 `design.json`。
- ABD-only 路径内部仍会生成 `design.json`。
- 该路径比 A1 更受设计生成 agent 影响。

## 8. 辅助 B：模型库与样本资产路径

辅助 B 是主路径的数据资产与质量提升链路。

### 8.1 定位

辅助 B 不直接从新需求生成柜。它负责维护主路径所需的数据底座：

- 模型库商品。
- 模型参数。
- 模型预览图。
- image profile。
- 已有组合柜样本。
- 参数关系报告。
- 回归 case。

### 8.2 基础流程

```text
线上模型库 / 收藏夹 / 组合柜样本
  -> 拉取商品列表
  -> 拉取模型参数
  -> 下载预览图
  -> 生成 image profile
  -> 构建单元柜池
  -> 分析组合柜参数关系
  -> 写入本地缓存
  -> 反哺主路径
```

### 8.3 输出资产

| 资产 | 用途 |
| --- | --- |
| `parammodel.json` | 商品列表，用于知道有哪些模型可选 |
| `parammodel_param_list.json` | 模型参数，用于脚本生成和参数绑定 |
| `parammodel_image_profile.json` | AI 选型检索依据 |
| 预览图 | 确认页展示和人工判断 |
| 组合柜样本 | 分析模板和沉淀回归 case |
| 参数关系报告 | 提炼父子参数绑定规则 |
| 回归 case | 验证主路径改动是否退化 |

### 8.4 辅助 B 与主路径的关系

主路径依赖辅助 B：

- 没有商品列表，主路径不知道有哪些模型。
- 没有 image profile，AI 只能凭名称猜选型。
- 没有参数列表，脚本生成难以设置子模型参数。
- 没有样本和回归，质量无法持续验证。

主路径反哺辅助 B：

- 选型错误 -> 修正 profile 或补候选模型。
- 无合适模型 -> 同步新目录或补模型库。
- bbox 经常失败 -> 补充参数规则和回归 case。
- 某类柜稳定失败 -> 进入样本分析和模板沉淀。

### 8.5 辅助 B 扩展：已有组合柜一键复建

辅助 B 可以扩展为“已有组合柜复建入口”。

流程：

```text
收藏夹/线上组合柜
  -> 导出 abd.json + assembly.json + parammodel_param_list.json + 封面图
  -> 写入智能体工作区
  -> 进入 ABD-only 复建路径
  -> 生成 design/script
  -> 编辑器执行
```

这不是从 0 生成新柜，而是将已有组合柜转为可复用、可验证、可沉淀的结构化输入。

## 12. MVP 验收标准

MVP 可验收条件：

1. 用户输入文字/图片后，系统可生成 `abd.json` 草稿。
2. 用户可在确认页完成单元柜和布局确认。
3. 确认后的 `abd.json` 可生成 `design.json`。
4. `design.json` 可生成 `cabinet_script.js`。
5. 脚本可通过 MCP 执行到参数化编辑器。
6. 系统可输出实例数量校验和 bbox 校验结果。
7. bbox 不一致时可自动重试，最多 2 次。
8. 失败时可保留中间产物并生成诊断报告。
9. 至少支持一批稳定 regression case。
10. 本地模型库至少包含商品、参数和 image profile 三类缓存。

## 13. 风险与问题

### 13.1 当前风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| ABD 契约新旧并存 | agent 输出不稳定 | PRD 和工程中统一为扁平 `abd.json` |
| `model-builder.md` 缺失 | agent 文档链路不完整 | 补齐或从 AGENTS.md 移除 |
| image profile 覆盖不足 | 选型准确率低 | 将 profile 维护纳入辅助 B P0/P1 |
| 参数模板来源不统一 | design 生成不稳定 | 明确父参数标准来源和默认规则 |
| 只确认布局不确认内部结构 | 用户可能误解能力边界 | 在 PRD 和 UI 文案中明确确认页定位 |
| MCP 依赖编辑器连接 | 执行链路可能中断 | 启动器和错误提示需产品化 |

### 13.2 待定问题

1. MVP 首批支持的组合柜类目是否只限定“一柜到顶衣柜”。
2. `abd.json` 是否需要保留 `candidates` 作为长期字段。
3. 确认页是否需要展示候选模型推荐原因。
4. 是否要将样本一键复建纳入 v0.1，还是放入 v0.2。
